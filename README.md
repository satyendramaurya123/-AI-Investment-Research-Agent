# InvestIQ — AI Investment Research Agent

A multi-agent AI system that produces analyst-grade investment research on any public company. Enter a company name and get a full **INVEST / HOLD / PASS** recommendation, backed by a financial analysis, a business/SWOT analysis, and a risk assessment — synthesized by a "CIO" decision agent, in under 30 seconds.

---

## Overview

InvestIQ takes a single input — a company name — and runs it through a 5-node AI pipeline:

1. Resolve the ticker and pull a full company/financial data snapshot
2. Three specialist agents (Financial, Business, Risk) analyze that snapshot **in parallel**
3. A fourth "Decision" agent (acting as CIO) synthesizes all three reports into one weighted recommendation

The output is a full report: overall score, INVEST/HOLD/PASS call with a confidence %, pros/cons, a SWOT grid, risk gauges across 6 dimensions, and a 5-year revenue chart — plus PDF/JSON export, search history, and bookmarks (all client-side via `localStorage`, no database).

There are no external financial or news data providers — every number and narrative in the report is generated by an LLM (Llama 3.3 70B, served by Groq) reasoning over its own training knowledge. That is a deliberate trade-off explained below.

## How to Run It

### Prerequisites
- Node.js 18+
- A free Groq API key — [console.groq.com/keys](https://console.groq.com/keys) (no credit card required)

### Setup

```bash
# 1. Install dependencies
npm install

# 2. Add your API key
# Open .env.local and set:
GROQ_API_KEY=your_groq_key_here

# 3. Run the dev server
npm run dev
```

Open [http://localhost:3000](http://localhost:3000), type a company name (e.g. "Apple", "Tesla", "Microsoft") and hit Analyze.

### Environment Variables

| Variable | Required | Notes |
|---|---|---|
| `GROQ_API_KEY` | Yes | Free tier: 1,000 requests/day on `llama-3.3-70b-versatile`. Get one at console.groq.com/keys |
| `NEXT_PUBLIC_APP_URL` | No | Used only for building share links; defaults to `http://localhost:3000` |

### Production build

```bash
npm run build
npm start
```

### Deploying

Deployed on **Vercel** (recommended for Next.js): push to GitHub → import the repo at vercel.com/new → add `GROQ_API_KEY` under Environment Variables → Deploy.

## How It Works

### Architecture

```
User enters company name
        │
        ▼
POST /api/analyze  (55s hard timeout — see trade-offs)
        │
        ▼
┌───────────────────────────────────────────┐
│ 1. collectDataNode                        │
│    ONE Groq/Llama call returns: ticker,   │
│    company profile, stock quote, 20+      │
│    financial metrics, 5yr revenue history │
└───────────────────────────────────────────┘
        │
        ▼
┌─────────────┬─────────────┬───────────────┐
│ 2. Financial │ 3. Business │ 4. Risk       │  ← run concurrently
│    Agent     │    Agent    │    Agent      │    via Promise.all
│ (6 dims,     │ (SWOT,      │ (6 risk       │
│  0-100 score)│  moat, 0-100)│  dims, 0-100)│
└─────────────┴─────────────┴───────────────┘
        │             │              │
        └─────────────┼──────────────┘
                       ▼
        ┌──────────────────────────────┐
        │ 5. Decision Agent (the "CIO") │
        │ Weighted score → INVEST/HOLD/ │
        │ PASS + confidence + reasoning │
        └──────────────────────────────┘
                       │
                       ▼
             AnalysisResult JSON
                       │
                       ▼
        React result page (charts, SWOT,
        risk gauges, export, warning banner
        if the LLM call failed)
```

Each of the 5 nodes is a plain async function that reads and returns a shared `state` object — there's no `@langchain/langgraph` `StateGraph` involved (see trade-offs). Nodes 2–4 are the only ones that can safely run in parallel, since none of them depend on each other's output — only the Decision agent needs all three.

### Prompting approach

Every agent is a single LangChain `ChatGroq` call with:
- A **system prompt** defining the agent's persona (e.g. "You are a senior equity research analyst...") and its exact JSON output contract
- A **human prompt** containing the relevant data (financial metrics, company profile, etc.) formatted as readable context
- A hard instruction to return **only JSON, no markdown** — parsed with a small `askLLM<T>()` helper that strips code fences and finds the outermost `{ ... }` before `JSON.parse`

The Decision agent's prompt is the most interesting one: it's given the suggested weighted score in the prompt itself (see below) but is explicitly told it *may agree or adjust* — so it's not just doing arithmetic, it's making a judgment call informed by, but not bound to, the formula.

### Scoring formula

```
Overall Score = (Financial × 0.40) + (Business × 0.35) + (Risk Safety × 0.25)

INVEST → Overall ≥ 65
HOLD   → Overall 40–64
PASS   → Overall < 40
```

### Failure handling

Every agent call is wrapped in try/catch with a hand-written fallback object (neutral score of 50, "N/A" fields) so a single failed call never crashes the whole request. But if **every** agent falls back, that's a signal something is fundamentally wrong (bad API key, exhausted quota, model retired) — not a real analysis. The pipeline detects this (`allFellBack` / `anyFellBack` in `investmentGraph.ts`) and attaches a `warning` field with the *actual* underlying error to the response, which the UI renders as a visible red banner instead of silently showing a generic-looking "HOLD, 53/100" that could be mistaken for a real result.

## Key Decisions & Trade-offs

| Decision | Why | What it costs |
|---|---|---|
| **One LLM, zero external data APIs** | Simplest possible setup (one API key), and Groq's free tier is generous enough to make this usable. | Every number is AI-estimated from training knowledge — not real-time. A demo/take-home tool, not a production trading terminal. |
| **Groq (Llama 3.3 70B) over Gemini** | I originally built this on Gemini 2.5 Flash. Gemini's free tier turned out to cap at **20 requests/day/model** for my project, which one single analysis (originally ~9 calls) could exhaust almost immediately — every subsequent analysis silently fell back to identical generic data. Groq's free tier is 1,000 req/day on `llama-3.3-70b-versatile`, ~50x more headroom, with no billing/card requirement. | Llama 3.3 70B is a capable model but generally reasons a notch below Gemini 2.5 Flash / GPT-4-class models on nuanced financial judgment. |
| **Consolidated data-collection into 1 LLM call (not 5)** | Originally split into separate calls for ticker lookup, profile, quote, metrics, and 5yr history — 5 calls before any analysis even started. Merged into one structured JSON call to conserve daily quota headroom (~9 calls/analysis → 5). | Slightly larger, more complex single prompt; harder to unit-test one field in isolation. |
| **3 parallel agents, not sequential** | Financial/Business/Risk agents don't depend on each other's output, only the Decision agent does. Running them via `Promise.all` cut wall-clock time by roughly 3x versus one-after-another. | Triples instantaneous concurrent request load against the daily quota at that moment — acceptable trade since the daily cap (not per-minute) is the binding constraint. |
| **No News/sentiment agent** | Originally had a 5th agent doing news sentiment analysis. Removed it: with no real news API, it was Groq/Gemini *inventing* plausible-sounding headlines — which isn't useful signal and risks looking like real news to an unsuspecting user. Better to not have a feature than to have a fabricated-content feature. | Lose one signal in the decision (weights redistributed: Financial 40% / Business 35% / Risk 25%, was 35/30/25/10 with News). |
| **No Compare-two-companies feature** | Originally built a `/compare` page running two full analyses (10 LLM calls) either in parallel or sequentially. On the 20-req/day Gemini tier this reliably exhausted quota mid-comparison and returned near-identical fallback data for both companies, which looked like a data bug rather than a quota bug. Rather than ship a feature that intermittently produces misleading identical results, I removed it. | Side-by-side comparison is a genuinely useful feature I'd like back — see "What I'd improve." |
| **Manual pipeline, not `@langchain/langgraph` `StateGraph`** | The control flow here is simple enough (5 nodes, one fan-out/fan-in) that a hand-written async pipeline is easier to read and debug than a graph library, and it avoids an extra dependency. | Lose LangGraph's built-in features (streaming, checkpointing, visual graph, conditional edges) if the pipeline grows more complex later. |
| **Hard 55s request timeout via `Promise.race`** | Free-tier LLM calls can occasionally hang or retry-loop; without a cap the frontend spinner could spin forever. | If a genuinely slow-but-working analysis takes >55s, it's cut off with a timeout error rather than eventually succeeding. |
| **localStorage for history/bookmarks, no backend DB** | Zero infra, zero auth, works immediately on any deploy. | No cross-device sync; clearing browser storage clears everything. |
| **Browser print-to-PDF instead of a PDF library** | No server-side rendering dependency (e.g. Puppeteer) needed. | Less polished than a purpose-built PDF, and depends on the user's print dialog. |

### Ambiguities I resolved myself
- The task didn't specify a particular LLM provider or a particular set of "agents" — I cho0se 4 (Financial / Business / Risk / Decision) as a reasonable minimal analyst team, and explicitly designed the Decision agent as a synthesizer rather than just an averager.
- No spec for what "risk" should quantify — I settled on 6 dimensions (competition, debt, regulatory, geopolitical, market, operational) as a standard sell-side risk taxonomy.
- No spec for real-time vs AI-estimated data — given the take-home nature of this and free-tier constraints, I chose AI-estimated and made that limitation explicit in the UI/README rather than pretending it's live market data.

## What I Would Improve With More Time

1. **Bring back a real News agent — properly.** Wire up a real news API (NewsAPI/GNews) instead of LLM-invented headlines, so the sentiment signal is grounded in actual events rather than plausible-sounding fiction.
2. **Bring back Compare, with per-company caching.** Cache each company's analysis (even a short-lived in-memory/KV cache) so comparing two companies you've *already* analyzed costs zero extra LLM calls instead of doubling quota usage.
3. **Real financial data source.** Wire in a real market-data API (e.g. Financial Modeling Prep, Finnhub) for the objective numbers (price, market cap, revenue) and reserve the LLM for what it's actually good at — qualitative synthesis and judgment — rather than inventing plausible financial statements.
4. **Migrate to a real `StateGraph`** (`@langchain/langgraph`) once the pipeline has more than one fan-out stage, to get streaming intermediate results to the UI (so the loading screen can show real progress instead of a simulated one) and built-in retry/checkpoint semantics.
5. **Cross-device history/bookmarks** via a lightweight backend (e.g. Supabase) instead of localStorage, so a saved analysis follows the user across devices.
6. **Confidence calibration.** Right now "confidence" is whatever number the LLM decides to output; I'd want to validate/calibrate this against how often INVEST calls at a given confidence actually look good in hindsight, even informally.
7. **Rate-limit-aware queueing.** Instead of a single 55s hard timeout, detect a 429 specifically and queue/retry the request with backoff and a progress message ("waiting for API quota, retrying in Ns...") instead of failing outright.
8. **Automated tests.** Currently validated via `tsc --noEmit` + `eslint` + manual runs; I'd add unit tests for the scoring/weighting math and the JSON-parsing fallback logic, which are the parts most likely to silently regress.
